# 量化交易系统需求

## 核心交易策略 (Strategy Details)

### 交易频率

是高频交易、日内超短线、日间波段，还是长线投资？

**解释**: 逻辑通常集中在寻找“高动量”或“均值回归”机会。

**选择**: 日内超短线。

### 选股逻辑

基于什么指标？（如：技术指标、财务数据、宏观指标、情感分析或机器学习模型）。

**解释**:

| 逻辑类型 | 优势 | 劣势 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **技术指标** (如 RSI, MACD) | 计算简单、响应快、易于回测。 | 信号滞后，在震荡市中容易被反复打脸。 | 趋势确认、超买超卖判断。 |
| **量价因子** (成交量, 换手率) | 真实反映资金动向，是超短线的核心。 | 数据噪音大，需要精细化过滤。 | 判断突破有效性、资金流入。 |
| **机器学习** (随机森林, LSTM) | 能捕捉非线性关系，自动挖掘复杂特征。 | 容易过拟合（回测好实盘差），解释性差。 | 复杂模式识别、价格预测。 |
| **情感分析** (新闻、社交媒体) | 捕捉市场热点，抢占先机。 | 数据处理难度极高，噪音极多。 | 热点题材炒作。 |

**选择**: 技术指标

### 买卖信号

具体的入场和离场规则是什么？（例如：MACD金叉买入，RSI超卖卖出）。

**解释**:

*   **动量突破 (Momentum)**：
    *   *优*：顺势而为，一旦抓住主升浪获利丰厚。
    *   *劣*：假突破多，回撤可能较大。
*   **均值回归 (Mean Reversion)**：
    *   *优*：胜率通常较高。
    *   *劣*：在单边行情中可能不断“接飞刀”，导致巨大亏损。

*   **多策略实现方式**：
    1. 逻辑组合 (AND/OR)：比如，只有当“动量突破”信号出现，且“RSI指标”显示尚未超买（均值回归的辅助判断）时才入场。这能提高胜率，但会减少交易机会。
    2. 信号评分制：系统同时运行 5 个策略，每个策略给出一个推荐仓位分（0-1）。系统汇总这些分数，分数越高，建议投入的资金越多。
    3. 并行独立运行：每个策略独立运行，各占 50% 的资金。


**选择**: 动量突破(先实现单策略)

### 回测需求

是否需要回测历史数据？需要多长时间范围、什么时间粒度（分钟级、日线级）的数据？

**选择**: 日线级回测(数据量小)，回测5年

## 数据源需求 (Data Sources)

### 数据提供商

你是否有偏好的数据源？（如：Tushare, AkShare, Baostock, JoinQuant, 或者付费的通达信、Wind 数据）。
介绍优劣

**解释**:

| 数据源 | 优势 | 劣势 |
| :--- | :--- | :--- |
| **AkShare** (推荐) | **完全免费**，接口丰富（含股票、基金、期货），无需注册 API Key。 | 抓取自网页，稳定性偶尔受原网站影响。 |
| **Tushare** | 数据质量高，格式规范，文档齐全。 | 实行积分制，部分高级数据（如日内分钟线）需要付费或贡献代码获取。 |
| **Baostock** | 免费、无需注册，提供基础的 K 线和财务数据。 | 接口较少，数据更新速度相对稍慢。 |

**选择**: 
* AkShare
* 应用启动时下载从上次启动后缺失的数据，增量下载
* 仅在需要更新当天数据时才访问 API
* 保存为Parquet 格式


### 数据类型

除了基础的行情数据（开高低收量），是否需要 Level-2 数据、基本面数据或新闻资讯？ 

**选择**: 基础的行情数据

## 风险控制与资金管理 (Risk Management)

### 仓位管理

是全仓进出，还是固定比例仓位？或者是基于凯利公式、马丁格尔等算法动态调仓？

**选择**: 
* 给出仓位建议: 虽然不直接连券商下单，但系统在回测或实时监控时，会输出类似这样的信息：
    * [信号生成] 2026-01-10 15:00:05 标的: 沪深300ETF (510300) 动作: 买入 建议仓位: 50% (当前可用资金的比例) 原因: 触发均线金叉 + 动量突破

### 止损止盈

是否有明确的硬止损位、追踪止损位或目标止盈位？

**解释**:

*   **止损策略**：
    *   *固定比例止损*：跌破买入价 N% 立即离场（简单有效）。
    *   *ATR 吊灯止损*：根据波动率动态调整止损位，避免被随机波动洗出。
*   **止盈策略**：
    *   *阶梯止盈*：达到目标 1 减半仓，目标 2 清仓。
    *   *移动止盈 (Trailing Stop)*：从最高价回落 N% 则清仓，锁定利润。

**选择**: 可以选择止损策略，先实现固定比例止损

### 最大回撤控制

对系统的最大风险承受能力是多少？

**解释**:

*   **风控开关**：当总账户回撤达到 X% 时，系统自动停止所有新开仓，强制进入休整期。

**选择**: 做成输入选项

## 执行与交易 (Execution)

### 运行模式

是仅提供交易信号（手动执行），还是全自动实盘交易？

**解释**: 个人用户能自动交易方式及限制:

1.  **直连 API**：国内主流券商（如中信、华泰）的官方 API 通常仅开放给资产在 **500万以上** 的机构或高净值个人。
2.  **三方软件 (推荐方案)**：
    *   **QMT (迅投)** / **PTrade (恒生)**：这是目前个人交易者最主流的选择。
    *   **前置条件**：在特定券商（如国金、中信建投等）开户，资产通常要求 **10万-50万** 即可免费申请使用。
    *   **费用**：通常仅收正常的交易佣金，无额外软件费。
3.  **非官方工具**：如 `easytrader`（模拟客户端操作），风险较高，容易被券商封禁风控，不建议。

**选择**: 先实现交易信号

如果是实盘，你打算使用哪个券商的接口？（注意：A股自动化交易接口通常有较高门槛）。

### 运行环境

系统需要部署在本地电脑、云服务器，还是作为轻量级脚本运行？

**选择**: 轻量级脚本运行，本地部署


### 框架偏好

是否需要基于现有的量化框架（如 Backtrader, PyAlgoTrade, QMT, PTrade）开发，还是从零构建？
介绍各类技术框架

**解释**: 框架选择:
*   **Backtrader**：Python 界最知名的回测框架。功能极其强大，支持多品种、多策略、复杂头寸管理，但学习曲线略陡峭。
*   **QMT/PTrade**：券商提供的内置框架。优点是策略写完直接能实盘交易，缺点是回测功能不如 Backtrader 灵活。
*   **从零构建**：如果你想完全掌握底层逻辑且策略不复杂，可以基于 `Pandas` 手写简单的回测逻辑。

**选择**: 从零构建,先简单点

### 展示方式

需要命令行输出、日志记录，还是需要一个可视化看板（如使用 Streamlit, Dash 或 Grafana）？

**解释**:
*   **命令行/日志**：适合开发调试，观察信号触发。
*   **Matplotlib/Plotly**：适合在回测结束后生成净值曲线图。
*   **Streamlit** (推荐)：如果你想要一个简单的网页看板。它能用纯 Python 快速写出带图表、参数调节滑块的交互界面。

**选择**: Streamlit
